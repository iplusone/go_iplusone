# Week 9: HTTP と JSON を、ただ返すのではなく構造として理解する

Web API を書き始めると、最初は「JSON を返せればOK」という気持ちになりやすいです。実際、Go の `net/http` と `encoding/json` はかなり素直なので、最小構成の API はすぐ書けます。

でも、そこから先で差が出るのは、HTTP をどれだけブラックボックスにしないかだと思っています。

Week 9 では、`net/http`、ミドルウェア、`encoding/json` を使って JSON API を作りながら、通信の構造を理解していきます。

まず `net/http` はとても強い標準ライブラリです。フレームワークがなくても、Handler を定義するだけでサーバが立ちます。これは Go の気持ちよさのひとつでもあります。

でも、ここで大事なのは「簡単に立つこと」よりも、「リクエストとレスポンスがどう流れているか」が見えやすいことです。

HTTP はリクエストメソッド、パス、ヘッダ、ボディ、ステータスコードといった要素の組み合わせで成り立っています。ここを曖昧にしたままフレームワークだけ使うと、後で問題が起きたときに何が起こっているか追えなくなります。

ミドルウェアも同じです。

ログ出力や認証、計測のような横断的な処理は、ルーティング本体に直接書くよりも、Handler をラップする形で分離した方が読みやすくなります。これは単におしゃれな設計ではなく、責務分離の基本です。

JSON についても、単に `Marshal` / `Unmarshal` を使うだけで終わらせず、ストリームとして扱えることを理解しておくとかなり強いです。巨大な JSON を毎回全部メモリに読み込む必要はありません。`json.Decoder` を使えば、`io.Reader` からそのまま順に読めます。

こういう話は一見細かいですが、HTTP と I/O とメモリの話がここでつながります。Week 5 のストリーム処理が、ここでそのまま活きてくるわけです。

この週が終わったときに答えられればよい問いは次の3つです。

- `net/http` で最小の JSON API をどう組むか
- ミドルウェアは何のために分けるのか
- なぜ巨大な JSON はストリームとして扱う方がよいのか

Week 9 は、Go で Web を書けるようになる週でもありますが、それ以上に「通信の構造が見えるようになる週」だと思っています。

## ここから有料

この週で大事なのは、HTTP を単なるルーティングの仕組みとして理解しないことです。

実際には、HTTP は異なるシステム同士が会話するための約束です。だから、メソッドやステータスコードを雑に扱うと、クライアント側との契約が曖昧になります。

また、ミドルウェアの価値は「便利だから」ではなく、「関心事を横に切り出せるから」です。ログや認証を各ハンドラに埋め込み始めると、すぐに見通しが悪くなります。

JSON の扱いでは、構造体タグだけでなく、「どこで読み、どこで検証し、どこで変換するか」を意識できるとかなり実務的になります。

この週を終える目安は、「HTTP ハンドラの中で何をやるべきで、何を外へ逃がすべきか」を考えられることです。
