# Week 6: 「動いているはず」を卒業して、正しさと性能を測る

実装をしていると、「たぶん動いている」「前に試したときは大丈夫だった」という感覚に頼ってしまうことがあります。

小さなコードならそれでも進めます。でも、少しずつ機能が増えていくと、その感覚はすぐに危うくなります。修正したら別のところが壊れた、速いと思っていた実装が実は遅かった、アロケーションが増えていたのに気づかなかった。そういうことが普通に起きます。

Week 6 では、それを防ぐために、テスト、ベンチマーク、エスケープ解析をまとめて扱います。

この週のテーマは、とてもシンプルです。`思い込みではなく、測って判断する`。

まずテストです。Go のテーブル駆動テストは、かなり実用的な書き方です。入力と期待値を並べてループで回すだけなので、ケース追加のコストが低く、境界値も整理しやすい。実務でもそのまま使いやすい型です。

大事なのは、テストを書くこと自体よりも、「何が壊れたら困るか」を言語化することです。正常系だけでなく、空入力、境界値、異常系も含めて振る舞いを固定していく。この感覚があると、コードはかなり安心して触れるようになります。

次にベンチマークです。

速いか遅いかは、案外主観で判断してしまいがちです。でも、主観はよく外れます。だから `go test -bench` でちゃんと測る。`ns/op` `B/op` `allocs/op` といった数字を見る。これだけでも、性能の会話がかなり具体的になります。

特に Go ではアロケーション回数が見えるのが面白いところです。単に処理時間だけでなく、どれだけメモリ確保しているかが分かるので、「なぜこの実装が重いのか」を考える入口になります。

さらに `-gcflags=-m` によるエスケープ解析を見ると、どこで値がヒープへ逃げているかが見えてきます。

もちろん、毎回これを細かく追う必要はありません。でも、ヒープ確保が増えると GC 負荷が上がりやすい、という感覚を一度コードと結びつけて持っておくと、その後の設計判断がかなり変わります。

Week 6 は、テストの書き方を学ぶ週でもあり、性能の見方を学ぶ週でもありますが、もっと根っこには「実装を観測する態度」を作る週だと思っています。

動いたから終わりではなく、どう動くか、どれくらい速いか、どこでメモリを使うかまで見に行く。これができるようになると、Go のコードを読む目が一段深くなります。

この週が終わったときに答えられればよい問いは次の3つです。

- テーブル駆動テストはなぜ整理しやすいのか
- `-benchmem` の数字は何を表しているのか
- なぜヒープ確保が増えると遅くなりやすいのか

## ここから有料

この週で大事なのは、「テストは安心のため」だけで終わらせないことです。実際には、テストは仕様を固定するための文章でもあります。

ケースをテーブルとして並べると、「この関数にはこういう入力が来る」「ここが境界だ」という前提が可視化されます。これはチーム開発でもかなり効きます。

ベンチマークでは、1回の数値だけで一喜一憂しないことも重要です。ローカル環境のばらつきはあるので、比較したい実装同士を同じ条件で測ることに意味があります。

また、速さだけを見てアロケーションを無視すると、本質を外すことがあります。Go では GC と仲良く付き合う必要があるので、`allocs/op` はかなり重要な手がかりです。

この週を終える目安は、「速そう」「大丈夫そう」ではなく、「この数字があるからこう判断した」と言えるようになることです。
