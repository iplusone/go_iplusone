# Week 5: I/O を「読む処理」ではなく「資源を扱う処理」として考える

Week 5 では、少し視点を変えます。

ファイルを読む、ネットワークへ送る、タイムアウトをつける。これらは一見すると別々の話に見えます。でも実際にはどれも、「外部リソースをどう扱うか」というひとつのテーマでつながっています。

Go のよいところのひとつは、I/O の抽象化がとても素直なことです。`io.Reader` と `io.Writer` という小さな interface が中心にあって、その上にファイル、ソケット、HTTP ボディ、バッファなど、いろいろなものが乗ります。

この週で学びたいのは、単にファイルが読めるようになることではありません。もっと大事なのは、「外部リソースは借り物であり、借りたら返す」「大きなデータは全部を一度に抱えない」「終わるべき処理は終わらせる」という感覚です。

巨大なデータを一括で読み込めば、それだけメモリを使います。短いサンプルでは問題なくても、サイズが大きくなると急に苦しくなる。そのときに必要になるのがストリーム処理です。

`io.Reader` / `io.Writer` を使えば、一定サイズのバッファで順に処理を流せます。これは派手な最適化ではなく、I/O を扱う上での基本姿勢に近いものです。

もうひとつ大事なのが `defer` です。

ファイルを開いたら閉じる。`rows` を取得したら閉じる。レスポンスボディを読んだら閉じる。こういう「最後に必ずやるべき処理」は、意識だけで守ろうとすると漏れます。

だから Go では、開いた直後に `defer Close()` を置く形がとても自然です。これは文法の小技ではなく、資源リークを防ぐ設計パターンです。

さらに `context` が入ってくると、「終わらない処理を作らない」という視点が加わります。HTTP でも DB でも外部 API でも、待ち続ける処理はシステム全体をじわじわ壊します。`context.WithTimeout` は、そのリスクをコードに持ち込まないための道具です。

この週を通して身につけたいのは、I/O の書き方そのものよりも、「資源には寿命がある」という感覚です。

ファイルはいつか閉じる必要がある。リクエストはいつか諦める必要がある。読み込みはいつか区切る必要がある。このライフサイクル意識があると、後で HTTP サーバや DB アクセスを書くときにもコードがかなり落ち着きます。

この週が終わったときに答えられればよい問いは次の3つです。

- なぜ巨大ファイルを一括読み込みしない方がよいのか
- なぜ `defer` を開いた直後に置くのか
- なぜ `context.WithTimeout` が必要なのか

Week 5 は、Go の I/O を覚える週というより、「ちゃんと終わるコード」を書く感覚を育てる週です。

## ここから有料

この週で大事なのは、「動くこと」と「安全に動き続けること」を分けて考えることです。

たとえばファイルを開いて読めたとしても、閉じ忘れていれば長く動いたときに資源を枯らします。HTTP リクエストを送れても、タイムアウトなしなら相手が止まったときに待ち続けます。

学習コードでは一回しか動かさないので見えにくいですが、実務ではこういう小さな漏れが積み重なって問題になります。

`defer` のコツは、「成功した直後に予約する」ことです。先延ばしにすると分岐が増えて漏れます。これをルール化してしまうと、かなり安定します。

また、`context` は「キャンセルの伝播」を持てるのが重要です。単にタイマーを持つだけなら別のやり方もありますが、親処理が終わったら子処理も終わらせる、という流れを自然に表現できるところが強いです。

この週を終えたと判断するポイントは、「I/O は処理ではなく資源管理でもある」と言えるようになることです。ここが分かると、HTTP、DB、運用の週がかなりつながります。
