# Week 11: 動くコードを、保ちやすい構造へ組み直す

ここまで来ると、Go でいろいろなものが作れるようになってきます。HTTP も、DB も、並行処理も、一通り触れてきました。

でも、実装が動くことと、実装が保ちやすいことは別です。

Week 11 では、レイヤ分離、エラー戦略、構造化ログを通して、「変更に強い構造」と「追跡できる実装」を作ることを考えます。

まずレイヤ分離です。

アプリケーションが大きくなるほど、入口、ユースケース、ドメイン、インフラの責務が混ざっていくと読みにくくなります。HTTP の都合、DB の都合、業務ロジックが同じ関数に入り始めると、変更のたびに影響範囲が広がります。

だから、層を分ける。`cmd`、`internal/domain`、`internal/usecase`、`internal/infra` のように役割を分けて、依存方向をできるだけ一方向に保つ。

ここで重要なのは、見た目をきれいにすることではありません。変更がどこまで伝播するかを抑えることです。

次にエラーです。

Go のエラー処理は地味ですが、とても重要です。`fmt.Errorf("...: %w", err)` のように文脈をつけてラップすることで、後から「どこで何が失敗したのか」をたどりやすくなります。

`errors.Is` や `errors.As` を使うと、見た目の文字列一致ではなく、意味としてエラーを判定できます。

そしてログです。

動いているコードは、問題が起きたときに外から見える必要があります。そこで構造化ログが出てきます。`log/slog` を使って JSON 形式で出しておくと、後で検索しやすく、集計しやすく、監視にもつなげやすい。

Week 11 のテーマは、「良い設計」の抽象論ではなく、「変えやすく、追いやすいコード」を作るための実践です。

この週が終わったときに答えられればよい問いは次の3つです。

- なぜ依存方向を一方向に保つべきなのか
- なぜエラーに文脈を足すのか
- なぜログを構造化するのか

ここまで来ると、単に Go が書けるのではなく、運用や保守を見据えて組めるようになってきます。

## ここから有料

この週で一番大事なのは、「層を分けること自体」が目的ではないと理解することです。目的は、変更しやすさと追跡しやすさです。

たとえばインフラの詳細がユースケース層へ漏れてくると、保存先変更のたびに業務ロジックまで触ることになります。逆に、境界がきちんと切れていれば、変える場所を限定できます。

エラーでも同じで、「failed」だけでは役に立ちません。どこで、何をしようとして、何が失敗したのかまで持っていないと、障害時に追えません。

構造化ログも、出しているだけでは不十分です。どのキーで後から絞り込みたいのか、何を関連づけたいのかを意識して設計すると、一気に実務感が出ます。

この週の確認ポイントは、「この情報がないと障害時に困る」という観点でコードやログを見られるようになることです。
