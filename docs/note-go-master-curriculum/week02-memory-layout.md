# Week 2: 値と参照とスライスを、文法ではなく挙動で理解する

Go を触り始めたとき、ポインタやスライスは「なんとなく使えるけれど、実は少し怖い」対象になりやすい気がします。

文法としてはそこまで難しくありません。`&` をつける、`*` で参照する、スライスは可変長の配列っぽいもの、といった説明でも最初は進めます。

でも、実装が大きくなってくると、それだけでは足りなくなります。なぜ値が書き変わらなかったのか、なぜ意図せず共有されたのか、なぜ `append` のあとに元の値と振る舞いが変わったのか。そういう場面で必要になるのは、文法知識よりも「実際に何がコピーされて、何が共有されているか」という感覚です。

Week 2 では、そこを正面から扱います。

この週のテーマは、値渡しと参照渡し、スライスの再確保、そして文字列のバイト表現です。一見するとバラバラに見えるかもしれませんが、全部「メモリ上でどう表現されているか」という共通の軸でつながっています。

たとえば値渡しでは、関数に渡した瞬間に何がコピーされるのかが重要です。整数のような単純な値なら、中身そのものがコピーされます。一方で、ポインタを渡せばコピーされるのはアドレスで、その先の実体は共有されます。

ここを理解していないと、「関数の中で変更したのに、なぜ元が変わらないのか」「なぜ勝手に変わったのか」が説明できません。

スライスも同じです。スライスは配列そのものではなく、内部的には「先頭へのポインタ」「長さ」「容量」を持っています。だから代入しただけでは実データが全部コピーされるわけではありません。見た目は値のようでも、内部には共有が潜んでいます。

さらに `append` が入ると、容量を超えたタイミングで新しい配列が確保されることがあります。ここがスライスをややこしく感じさせるポイントです。同じように見えるコードでも、再確保が起きたかどうかで共有関係が変わるからです。

この週でそれを実験して観測すると、「スライスは便利だけれど、内部構造を知らないと危ない理由」がかなり腹落ちします。

文字列についても、ここで一度ちゃんと見ておく価値があります。Go の `string` は文字の配列ではなく UTF-8 のバイト列です。なので `len("日本語")` は文字数ではなくバイト数を返します。

これも最初は「そういうもの」として覚えがちですが、後で JSON、HTTP、ファイル I/O、文字コードの問題に触れたとき、この感覚があるかどうかで理解の深さが変わります。

Week 2 の狙いは、ポインタやスライスを怖がらずに使えるようになることではありません。もっと正確に言うと、「どこに共有があり、どこでコピーが起き、どこで再確保が起きるかを意識できるようになること」です。

この視点があると、後で性能の話にもつながります。コピーが多いコードはなぜ重いのか。ヒープ確保が増えると何が起きるのか。なぜ巨大なデータを雑に持ち回さないほうがよいのか。そうした話の土台がここでできます。

この週が終わったときに答えられればよい問いは、次の3つです。

- 値渡しと参照渡しで、何がコピーされるのか
- `append` によるスライスの再確保はいつ起きるのか
- `len("日本語")` が `9` になる理由は何か

Week 2 は派手ではありません。でも、ここで「見た目」ではなく「内部挙動」で考える習慣がつくと、その後の設計もデバッグもかなり安定します。

## ここから有料

この週で特に大事なのは、「ポインタを使うかどうか」よりも「共有が起きているかどうか」を意識することです。

実際、ポインタを明示的に使っていなくても、スライスや map、チャネルのように、内部に参照的な性質を持つ値はいくつもあります。だから「ポインタを書いていないから安心」とは言えません。

値渡しで誤解しやすいのは、「関数に渡したから全部複製された」と思ってしまうことです。整数ならそれで近いですが、スライスでは事情が違います。スライス値そのものはコピーされても、その先の配列は共有されたままだからです。

この違いを理解するコツは、「コピーされたのは何の単位か」を分けて考えることです。

- 整数なら中身そのもの
- ポインタならアドレス
- スライスならヘッダ

この整理ができると、関数の中で要素を書き換えたときに外側へ影響が出る理由、出ない理由がかなり説明しやすくなります。

`append` の理解では、「容量を超えるまでは元の配列を使う」「超えたら別の配列になる可能性がある」という2段階で覚えると実用的です。ここが曖昧なままだと、バグの再現が不安定に見えることがあります。

文字列については、「画面に1文字に見えるもの」と「バイト列として何バイトか」は別の話だと切り分けるのが大事です。HTTP ボディや JSON、DB 保存では最終的にバイトとして扱う場面が多いので、ここを早めに知っておくと後が楽です。

この週を終える目安は、単に実験コードが動くことではなく、「この関数呼び出しでは何がコピーされるか」を、コードを見る前に予想できるようになることです。
