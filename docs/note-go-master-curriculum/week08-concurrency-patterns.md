# Week 8: 並行処理を安全に使うための基本パターンを持つ

Week 7 で goroutine、channel、select の原理を見たら、次はもう少し実戦寄りに進みます。

並行処理は便利ですが、便利だからこそ壊れやすいです。特に共有データが絡んだ瞬間に、見えにくいバグが入り込みます。

Week 8 では、Mutex、WaitGroup、Worker Pool という、Go で頻繁に出てくる基本パターンを扱います。

まず Mutex です。

複数の goroutine が同じ変数を同時に更新すると、`count++` のような一見単純な処理でも安全ではありません。読み取り、加算、書き戻しがひとまとまりの不可分操作ではないからです。

この事実は、最初は少しショックかもしれません。たった1行なのに安全ではない。けれど、ここを理解すると「並行処理では見た目よりも共有が危ない」という感覚が身につきます。

Mutex は、その共有区間を一度に1つの goroutine だけが通れるようにする道具です。地味ですが、かなり重要です。

次に WaitGroup です。

goroutine を起動しただけでは、いつ終わるか分かりません。処理の完了を待ちたいときには、その待ち合わせを明示する必要があります。WaitGroup はそのためのとても素直な仕組みです。

そして Worker Pool です。

goroutine が軽いとはいえ、無制限に起動すればよいわけではありません。特に I/O や外部 API、DB などが絡むと、相手側の容量や自分側のメモリも考える必要があります。

Worker Pool は、処理数ではなく並列度を制御するための基本パターンです。ジョブを channel で配り、固定数の worker が処理する。この構造を一度自分で書けるようになると、並行処理の実装がかなり整理されます。

Week 8 のテーマは、「並行処理を使う」から「並行処理を制御する」への一歩です。

この週が終わったときに答えられればよい問いは次の3つです。

- なぜ `count++` は安全ではないのか
- Mutex と WaitGroup の役割はどう違うのか
- Worker Pool は何を制御しているのか

ここが分かると、並行処理はかなり怖くなくなります。雑に増やすのではなく、どこを守り、どこを待ち、どこを制限するかを考えられるようになるからです。

## ここから有料

この週のポイントは、「並行処理の問題はスピードの問題である前に整合性の問題でもある」と知ることです。

速くしたいから goroutine を使う、という入り方は自然です。でも、共有メモリが壊れたら速さ以前に正しさが崩れます。だから Mutex が先に出てきます。

WaitGroup も、「終わるまで待つ」という単純な話に見えて、実際にはプログラムのライフサイクル制御の入口です。メイン処理が先に終わってしまえば、起動した goroutine は最後まで動けません。

Worker Pool では、「何件の仕事があるか」と「何個の worker を立てるか」を分けて考える感覚が重要です。ここが分かると、外部サービスへの負荷制御やバッチ処理の設計にもつながります。

この週を終えた判断基準は、「この並行処理コードは何を守っていて、何を待っていて、何を制限しているのか」を言えることです。
