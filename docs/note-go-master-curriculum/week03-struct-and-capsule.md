# Week 3: struct をデータの箱で終わらせず、境界として使う

Go を勉強し始めると、`struct` はかなり早い段階で出てきます。複数の値をひとまとめにできる便利な型、という理解でも最初は進めます。

でも実装が大きくなってくると、構造体は単なる箱ではなくなります。どんなデータを持つのか、どういう操作を許すのか、外からどこまで触らせるのか。つまり構造体は、データの集約だけでなく「境界」を作るための道具になります。

Week 3 では、その感覚を掘り下げます。

この週で扱うのは、構造体の物理配置、値レシーバとポインタレシーバの違い、そしてパッケージによるカプセル化です。

まず構造体の物理配置について。普段はあまり意識しませんが、フィールドの順序によって構造体のサイズは変わります。CPU が読みやすい配置にするために、コンパイラはアライメントやパディングを入れるからです。

この話は、日常的に毎回気にする必要があるわけではありません。ただ、「構造体は見た目通りにぎゅうぎゅう詰めで並ぶわけではない」という感覚を一度持っておくと、データがメモリ上にどう置かれるかへの解像度が上がります。

次にレシーバです。Go では、メソッドを値レシーバで持つか、ポインタレシーバで持つかを自分で選びます。

ここが大事なのは、単なる書き方の違いではなく、`コピーに対して振る舞うのか` `実体そのものに振る舞うのか` を決めている点です。状態を変更したいメソッドならポインタレシーバが自然ですし、コピーして扱っても問題ない不変的な振る舞いなら値レシーバでもよい。

この選択を「なんとなく」で済ませず、意図を持って決められるようになることが、この週の大きな意味です。

そしてもうひとつ、個人的に重要だと思っているのがカプセル化です。

Go はオブジェクト指向言語のような厳密なアクセス修飾子を持っていませんが、その代わりに、公開・非公開のルールがかなりシンプルです。大文字始まりは公開、小文字始まりは非公開。さらに `internal` ディレクトリでインポート範囲も制限できます。

このシンプルさは、最初は地味に見えます。でも、実装を守るという意味ではかなり強いです。

たとえば、外から勝手に値を書き換えられたくない構造体があるなら、フィールドを非公開にして、必要な操作だけメソッドとして公開する。その時点で、「どんな状態変化を許すか」を型の側でコントロールできます。

これは「きれいな設計」のためだけではありません。壊されにくいコードにするための実践的な手段です。

Week 3 の終わりでは、次のことを説明できれば十分です。

- なぜポインタレシーバを選ぶのか
- フィールド順序でサイズが変わるのはなぜか
- 非公開フィールドを持つ型にする意味は何か

構造体の週というより、「データに責任を持たせる週」だと思うと、このテーマはかなり理解しやすくなります。

## ここから有料

この週でつまずきやすいのは、「値レシーバかポインタレシーバか」を文法の話だけで考えてしまうことです。

実際には、ここで考えたいのは次の3点です。

- そのメソッドは状態を変更するか
- その値はコピーしても意味が崩れないか
- 将来的に大きな構造体になったとき、コピーコストを許容できるか

特に1点目はかなり重要です。状態を変更するのに値レシーバを使うと、「変わったつもりなのに変わっていない」コードが生まれます。これは実装初期では気づきにくく、あとで読む人をかなり混乱させます。

カプセル化についても、「外から触れないようにする」だけでなく、「変更の入口を絞る」ことがポイントです。たとえば `Count` フィールドを公開して自由に書き換えられるより、`Increment()` メソッドだけを公開したほうが、状態の変化を追いやすくなります。

この週の理解が深まると、後の DI やアーキテクチャの週でも、「何を公開し、何を隠すか」という感覚がかなり効いてきます。つまり Week 3 は、単なる struct の週ではなく、設計の入口の週でもあります。
