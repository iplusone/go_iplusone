# Week 7: goroutine と channel を、魔法ではなく仕組みとして理解する

Go を学ぶ理由のひとつとして、並行処理の書きやすさを挙げる人は多いと思います。実際、goroutine と channel の組み合わせはとても魅力的です。

ただ、書きやすいことと、理解しやすいことは少し別です。

`go` をつければ並行になる。channel で値を渡せる。`select` で複数待てる。そこまでは分かっても、「なぜそれで同期が取れるのか」「なぜ止まるのか」「なぜデッドロックするのか」まで含めて腹落ちしているかというと、ここは人によって差が出やすいところです。

Week 7 では、その基礎原理を扱います。

まず goroutine です。Go の並行処理は OS スレッドを直接増やしているわけではありません。goroutine はランタイムが管理する軽量な実行単位で、OS スレッドへの割り当てもランタイムが面倒を見ます。

このおかげで、かなりたくさんの goroutine を気軽に起動できます。ただし、「軽いから無限に雑に使っていい」わけではありません。軽さの背景にある仕組みを知っておくと、過信しにくくなります。

次に channel です。

個人的に、Go の並行処理を理解するときに一番大きいのは、「channel はデータ受け渡しの道具であると同時に、同期の道具でもある」という点だと思っています。

特に unbuffered channel では、送信側と受信側がそろうまで双方が待ちます。つまり値を渡しているように見えて、実際には「ここで待ち合わせる」という意味も持っています。この感覚が分かると、channel の見え方がかなり変わります。

そして `select` です。

複数の channel を同時に扱いたいとき、`select` はとても自然な書き方を提供してくれます。複数の待ち合わせポイントのうち、準備できたものをひとつ選ぶ。この考え方は、タイムアウトやキャンセルにもつながっていきます。

Week 7 の狙いは、並行処理をたくさん書けるようになることよりも、並行処理が「なぜそのように動くか」を説明できるようになることです。

この週が終わったときに答えられればよい問いは次の3つです。

- unbuffered channel が同期になるのはなぜか
- `select` は何をしているのか
- goroutine は OS スレッドと何が違うのか

ここが分かると、次の Week 8 で扱う Mutex や Worker Pool がかなり理解しやすくなります。

## ここから有料

この週の学習でありがちなのは、「goroutine を増やすと速くなる」と思い込むことです。でも、実際には I/O 待ちなのか CPU バウンドなのか、共有資源があるのか、待ち合わせが多いのかで話は変わります。

channel でも同じで、「値を渡せる」だけで終わると弱いです。重要なのは、送受信のタイミングそのものが同期点になっていることです。ここが分からないと、なぜ詰まるのか、なぜ進むのかが見えません。

`select` についても、「便利な switch みたいなもの」ではなく、「複数の通信路を待つための制御構文」として捉えるのが大事です。ここが分かると、`time.After` を絡めたタイムアウトやキャンセルも自然に読めるようになります。

この週を終えたときの理想は、「このコードはどこで待っているのか」を、並行処理コードを見ながら追えるようになることです。
