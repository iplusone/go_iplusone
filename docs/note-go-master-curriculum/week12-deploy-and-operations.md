# Week 12: 動かすだけで終わらず、止め方と配り方まで設計する

12週間の最後は、配布と運用です。

ここまで来ると、「Go で動くものを作る」こと自体はかなりできるようになっています。でも、本当にシステムとして考えるなら、最後に必要なのは「どう動かし続けるか」「どう安全に止めるか」「どう配るか」です。

Week 12 では、`scratch` ベースの極小イメージ、グレースフルシャットダウン、ヘルスチェックを扱います。

まず極小イメージです。

Go は静的リンクしやすいので、`CGO_ENABLED=0` と `scratch` を組み合わせることで、実行バイナリだけを含むかなり小さなイメージを作れます。これは単なるサイズ削減ではなく、攻撃面の縮小にもつながります。

次にグレースフルシャットダウンです。

アプリケーションは起動より停止のほうが軽視されがちです。でも実際には、止まり方が雑だと処理中のリクエストが切れたり、データの整合性を崩したりします。

Docker や Kubernetes は、停止時にいきなり破壊するのではなく、まず SIGTERM を送ります。その合図を受け取って、処理中のリクエストが終わるのを待ってから終了する。これがグレースフルシャットダウンです。

そしてヘルスチェックです。

`/healthz` のようなエンドポイントを持っていれば、外側のオーケストレータや監視が「今このプロセスは生きているか」を確認できます。これは地味ですが、運用可能性の入口です。

Week 12 のテーマは、「アプリケーションは起動した瞬間に完成するわけではない」という事実を受け止めることです。

配布しやすいか。停止時に安全か。外から見えるか。ここまで含めて、ようやくシステムとしてひとまとまりになります。

この週が終わったときに答えられればよい問いは次の3つです。

- `scratch` ベースイメージの意味は何か
- なぜ SIGTERM を受けてすぐ終了するのは危険なのか
- `/healthz` は何のためにあるのか

12週の最後にこのテーマを置いているのは、ここまで学んできたことが全部つながるからです。I/O の寿命管理も、HTTP サーバの構造も、context も、運用の週で回収されます。

## ここから有料

この週で大事なのは、「本番っぽいことを最後に少し触る」ではなく、「止め方も実装の一部だ」と認識することです。

学習段階では、Ctrl+C で落として終わりにしがちです。でも、本番ではその瞬間にも誰かのリクエストが流れているかもしれません。だから終了制御が必要です。

極小イメージについても、「小さいからえらい」で終わらせず、何が入っていて、何が入っていないかを意識することが重要です。不要なものがなければ、それだけ攻撃面も減ります。

ヘルスチェックは、生きているかどうかを外から見せる最小の窓です。これがあるだけで、監視やオーケストレーションとの接続がかなりやりやすくなります。

この週の完了ラインは、「アプリケーションをどう安全に終わらせるか」を、起動方法と同じくらい自然に話せることです。
