# Week 10: DB を「つながる箱」ではなく、有限資源と整合性の場として扱う

DB を使うコードは、最初は CRUD ができれば十分に見えます。`SELECT` できる、`INSERT` できる、更新できる。そこまでは比較的早く到達できます。

でも、実際に重要なのはその先です。

接続は有限資源であること。SQL は安全に組み立てる必要があること。複数更新は一貫して成功するか、全部戻るかのどちらかであるべきこと。つまり DB は、単にデータを置く場所ではなく、整合性を守るための重要な境界です。

Week 10 では、`database/sql`、プリペアドステートメント、トランザクションを扱います。

まず `database/sql` で大事なのは、`sql.Open` が単純な1接続ではなく、プールの入口だということです。DB 接続は無限ではありません。だから接続数やアイドル数を意識する必要があります。

この視点があると、DB が「ただつながればいいもの」ではなく、共有される外部資源だと分かります。

次にプリペアドステートメントです。

これはセキュリティのためだけではありませんが、まず一番分かりやすいのは SQL インジェクション対策です。値を文字列連結で埋め込むのではなく、プレースホルダと引数バインドで渡す。この分離はかなり重要です。

さらに、命令と値を分けることで、DB 側の実行計画キャッシュにもつながります。

そしてトランザクションです。

個人的には、この週の中心はここだと思っています。複数の更新が途中で失敗したとき、半分だけ反映されると整合性が壊れます。そういう状態を防ぐために、`Begin` `Commit` `Rollback` があります。

たとえば在庫を減らして注文履歴を追加する処理があったとして、片方だけ成功したら困ります。だから「全部成功するか、全部戻るか」を明示する必要があります。

この週が終わったときに答えられればよい問いは次の3つです。

- なぜプリペアドステートメントを使うのか
- `Begin` / `Commit` / `Rollback` は何を守っているのか
- なぜトランザクションなしでは整合性が壊れるのか

Week 10 は、DB を使う週というより、データの一貫性に責任を持つ週です。

## ここから有料

この週でありがちな誤解は、「DB に書けたらOK」と思ってしまうことです。でも、本当に重要なのは「失敗したときどうなるか」です。

成功時の処理は誰でも書けます。差が出るのは途中失敗の扱いです。だからトランザクションが出てきます。

また、`rows.Close()` や `context` 付きクエリのような周辺も地味に重要です。DB アクセスも I/O なので、寿命管理を怠ると後で効いてきます。

プリペアドステートメントは、「安全だから使う」で終わらせず、「命令と値を分離する設計」だと捉えると理解しやすいです。

この週を終える目安は、「この操作はなぜトランザクションで囲う必要があるのか」を、業務的な文脈で説明できることです。
