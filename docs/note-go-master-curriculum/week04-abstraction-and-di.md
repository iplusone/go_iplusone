# Week 4: interface と DI を「お作法」ではなく設計の道具として使う

Go を勉強していると、`interface` は早い段階から出てきます。でも、最初のうちは「とりあえず抽象化に使うもの」くらいの理解で通り過ぎがちです。

実際には、interface を使うべき場面と、使わないほうがいい場面があります。そしてその違いは、文法ではなく依存関係の設計にあります。

Week 4 では、interface と依存性注入を使って、上位のロジックを具体的な実装から切り離す考え方を扱います。

この週でやりたいのは、「とにかく interface を切る」ことではありません。むしろ逆で、どこを抽象化すると変更しやすくなり、どこは具体のままでよいのかを見極める感覚を作ることです。

たとえば、ビジネスロジックが直接 DB 実装に依存していると、テストのたびに DB が必要になったり、保存先を変えたいときにロジック側まで引きずられたりします。

そこで、保存や取得という振る舞いだけを小さな interface として定義し、ロジック側はその振る舞いだけを知るようにします。本番では実 DB を渡し、テストではメモリ上の Fake を渡す。これが依存性注入の基本的な形です。

Go の interface は暗黙的実装なので、「この型はこの interface を implements します」と明示的に書く必要はありません。必要なメソッドセットを満たしていれば、それで interface として使えます。

この特徴のおかげで、使う側から必要最小限の interface を定義しやすいのが Go のよいところです。

個人的には、この週で一番大事なのは「テストしやすいから interface を使う」のさらに手前にある考え方です。つまり、上位ロジックが「何をしたいか」を語り、下位実装が「どうやるか」を引き受ける、という責務分離です。

この分離ができると、コードはかなり読みやすくなります。ユースケースは振る舞いとして読めるし、インフラ層は具体的な保存方法として閉じ込められます。

もちろん、何でも interface 化すればよいわけではありません。Go では、必要になるまでは具体型のまま持つほうが分かりやすいことも多いです。それでもこの週を入れているのは、「どこを抽象化すると全体が楽になるか」を理解することが、後のアーキテクチャ設計に直結するからです。

この週の終わりでは、次のことを説明できれば十分です。

- なぜ interface を使うと実装差し替えやテストがしやすくなるのか
- コンストラクタ注入の形がなぜ自然なのか
- なぜ Fake 実装があると DB なしでロジックを検証できるのか

設計の話は、うまくいくと急にコードが読みやすくなります。Week 4 は、その最初の手応えを作る週です。

## ここから有料

この週でありがちな誤解は、「抽象化は正義だから最初から全部 interface にすべき」という考え方です。

でも実際には、小さなアプリや学習コードでは、具体型のまま持ったほうが分かりやすいことも多いです。interface が効くのは、「差し替えたい境界」がはっきりしているときです。

特に有効なのは次のような境界です。

- DB や外部 API など、外部環境に依存するもの
- テストでは置き換えたいもの
- 将来的に実装が複数ありうるもの

逆に、ドメインモデルや単純なヘルパーにまで過剰に interface を切ると、読む側の負荷が増えます。

また、Go では「使う側が interface を定義する」発想がかなり重要です。実装側が先に巨大な interface を作ると、依存が重くなりがちです。必要な振る舞いだけを小さく切る。ここが Go らしさのひとつです。

この週の確認観点としては、「このロジックが今どの具体実装に縛られているか」を見つけられるようになると強いです。縛られている場所が見えれば、そこが抽象化の候補になります。
