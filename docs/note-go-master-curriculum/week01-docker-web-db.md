# Week 1: Docker・Web・DB を最初にまとめて触る理由

Go の学習というと、最初は `Hello, World` から始めて、変数、関数、構造体、interface と少しずつ進んでいく流れが一般的です。もちろんその順番は自然ですし、必要でもあります。

ただ、実装の全体像を早い段階でつかむという意味では、最初にもう少し大きな絵を見ておいた方が後で効いてくることがあります。

このカリキュラムの Week 1 では、Go のコードを単体で動かして終わりにしません。`front` `app` `db` の3つを Docker Compose で立ち上げて、ブラウザからボタンを押し、Go の API を経由して、DB に入っているデータを画面へ表示するところまでを最初に確認します。

最初にこれをやる理由はシンプルで、この先に学ぶことが最終的にどこへつながるかを最初に見せたいからです。

ポインタの話も、I/O の話も、並行処理の話も、アーキテクチャの話も、単体で見ると抽象的になりがちです。でも、「最終的にはこういう形で Web アプリや API を作るために必要なんだ」と分かっていると、地味に見える基礎も意味を持ちます。

特に Go は、標準ライブラリだけでかなり広い範囲まで触れられる言語です。だからこそ、言語機能だけを順番に追うよりも、最初にシステムとしての輪郭を見せたほうが、後半の理解がつながりやすいと思っています。

この週で確認する流れは次の5つです。

1. Docker Compose で複数サービスを起動する
2. Go コンテナから DB コンテナへ接続する
3. ブラウザから Front コンテナへアクセスする
4. Front の JavaScript から Go API を呼ぶ
5. Go が DB から取得したデータをブラウザに表示する

ここで大事なのは、細かいコマンドを覚えることよりも、「それぞれが別の役割を持ったプロセスとして動いている」と理解することです。

ブラウザは DB を直接知らない。フロントは画面を持つ。Go は API として振る舞う。DB はデータを持つ。こうした役割分担を最初に見ておくと、その後の設計や分離の話がかなり飲み込みやすくなります。

この週で特に重要なのが、`localhost` の見え方です。

ホストマシンのブラウザから見る `localhost:18080` と、`app` コンテナの中から見る `localhost` は同じではありません。コンテナの中から `localhost` と言ったら、それはそのコンテナ自身を指します。だから Go から DB へ接続するときは `localhost` ではなく、Compose ネットワーク上のサービス名 `db` を使います。

この感覚は、最初はやや混乱しやすいです。でも、ここを曖昧にしたまま進むと、後から Docker や本番環境に触れたときに一気につまずきます。逆に、最初に「ホストの localhost」と「コンテナの localhost」は別物だと分かってしまえば、その後のネットワーク理解がかなり楽になります。

もうひとつ、この週で地味に大事なのが「初期状態を再現できること」です。

DB の初期データを `initdb/001_seed.sql` のようなファイルで入れておくと、環境を壊して作り直しても同じ状態から始められます。これは単なる便利機能ではなく、学習でも実務でもかなり重要です。手で作った状態に依存するものは、時間が経つほど再現しにくくなります。

なので Week 1 では、Go のコードそのものよりもむしろ、

- 複数サービスがいること
- それぞれがネットワークでつながること
- ブラウザは API に話しかけること
- DB の状態はコードや SQL で再現できること

この4つを自分の言葉で説明できるようになることが大切です。

この週が終わった時点で、次の問いに答えられれば十分です。

- なぜ Go から DB に `db:3306` でつなぐのか
- なぜブラウザから `localhost:18080` で画面が見えるのか
- なぜブラウザは DB ではなく Go API を呼ぶのか
- なぜ SQL 初期化ファイルがあると再現性が上がるのか

Week 1 は派手なことをしているようでいて、実は「今後の12週間の座標軸」を作っている週です。この座標軸があることで、後から出てくるメモリ、I/O、並行処理、設計、運用の話が全部ひとつの地図の上に乗ります。

## ここから有料

ここからは、Week 1 を実際に進めるときに混乱しやすいポイントを、もう少し具体的に整理します。

まず最初に押さえておきたいのは、「Docker Compose で3つ立ち上がる」という事実を、単なるコマンド結果で終わらせないことです。`front` `app` `db` が動いているなら、それぞれは別プロセスで、別の責務を持ち、別の失敗の仕方をします。ここを曖昧にすると、後で障害の切り分けができません。

次に、`front -> app -> db` という通信方向を頭の中で固定しておくと理解しやすいです。

- ブラウザは `front` の HTML と JavaScript を受け取る
- JavaScript は `app` の API を呼ぶ
- `app` は `db` と会話する

このとき、ブラウザが MySQL に直接つなぐわけではありません。ここを「画面から DB のデータを見ている」とだけ理解してしまうと、どこに責任があるのかが曖昧になります。画面はデータを直接持っているのではなく、Go の API が返した結果を見せているだけです。

また、`localhost` の混乱をほどくコツは、「誰の視点の localhost か」を毎回言葉にすることです。

- ブラウザ視点の `localhost:18080`
- ホストマシン視点の `localhost`
- `app` コンテナ視点の `localhost`
- `db` コンテナ視点の `localhost`

これらは全部別です。最初は面倒でも、この主語を毎回意識するだけで理解がかなり安定します。

学習の確認としては、次の観察をおすすめします。

- `docker compose ps` で3サービスが分かれていることを見る
- `docker compose logs app` で Go 側の接続先を確認する
- `docker compose exec db ...` で DB の中身を直接見る

ここまでやると、「ブラウザに表示されたもの」が本当に DB に由来していることが一本の流れとして見えてきます。

この週でありがちな誤解は、「動いたから分かったつもりになる」ことです。実際には、起動できることよりも、通信経路を説明できることのほうが大事です。

Week 1 の完了ラインは、次の一文を自分の言葉で話せるかどうかです。

「ブラウザはフロントにアクセスし、フロントの JavaScript が Go API を呼び、Go が DB から読んだ結果を返している。Go から DB へは localhost ではなく Compose のサービス名でつなぐ」

ここまで言えれば、この週は十分に前へ進めています。
