## 本文下書き

# Goの文法学習で終わらせない、12週間の実装技術カリキュラムを作った

最近、Go の学習コンテンツを整理しながら、ずっと引っかかっていたことがありました。

それは、Go の勉強をしているはずなのに、気づくと「文法だけが頭に残って、実装の感覚が残らない」という状態がかなり起きやすいことです。

たとえば `for` は書ける、`struct` も書ける、`interface` もなんとなく分かる。`goroutine` や `channel` という言葉も知っている。でも、じゃあ実際に Docker で立ち上がった Go アプリが DB と通信して、HTTP で JSON を返して、止め方まで含めてちゃんと運用できる形に持っていけるかというと、そこにはかなり大きな段差があります。

この段差は、単に学習量が足りないという話ではない気がしていました。むしろ、学ぶ順番や、知識のつなぎ方の問題が大きいのではないか、という感覚です。

Go の記事や教材はたくさんあります。良いものも多いです。ただ、読んでいて何度も感じたのは、文法、Web、DB、並行処理、運用がそれぞれ別の話として置かれていて、ひとつの学習体験としてつながっていないことでした。

もちろん、個別の記事としてはそれで正しいです。必要なテーマを個別に調べられるのは便利ですし、実際それに何度も助けられてきました。

でも、学ぶ側として見ると、便利さと引き換えに「今の自分は全体のどこにいるのか」がすごく見えにくくなります。メモリの話を読んだあとに、急に API の話を読んで、そのあと DB の接続プールの話に飛び、別の日にテスト戦略の記事を読む。どれも大事なのに、頭の中では一本の線になりにくい。

それで今回、Go の文法学習で終わらず、実装技術としての Go を一続きで学ぶための `Go実装技術マスター 12週間カリキュラム` を、自分なりに一本の流れとして整理しました。

これは、いわゆる「最短で覚える Go 入門」を作りたかったわけではありません。どちらかというと逆で、回り道に見えても、後半で崩れにくい土台を作ることを優先した構成です。

このカリキュラムの目的は、`Go の書き方を覚えること` ではありません。目指しているのは、`プログラミング全般の知見と計算機の仕組みを理解したうえで、Go で実装できるようになること` です。

少し大げさに聞こえるかもしれませんが、個人的にはこの違いはかなり大きいと思っています。Go を知っている状態と、Go でシステムを組み立てられる状態は、似ているようでかなり違います。

## このカリキュラムを作った理由

このカリキュラムを作ろうと思った理由は、単純に「教材を増やしたい」からではありませんでした。もっと根本的に、Go 学習の途中で起きやすい詰まり方を減らしたかった、という気持ちがあります。

Go 学習の情報は世の中にたくさんありますが、整理していく中で、次のような分断を何度も感じました。

- 文法学習はあるが、その先の実装につながりにくい
- Web、DB、並行処理、運用が個別トピックとして切り離されている
- 学ぶ順番が人によってばらばらで、全体像を見失いやすい
- 「動いた」で終わりやすく、「なぜ動くのか」まで残りにくい
- 入門のあとに、次にどこへ進めばよいかが見えにくい

特に気になっていたのは、「Go を勉強したはずなのに、システムの見え方があまり変わらない」という状態です。

たとえば `net/http` のサンプルを写経してサーバを起動できるようになっても、HTTP がどういう前提で動いているか、1 リクエストごとにどういう処理単位になるのか、巨大な JSON をどう扱うべきか、DB との境界をどう考えるべきか、という話までは自然にはつながりません。

ポインタやスライスもそうです。文法としては理解していても、実際に何がコピーされていて、どこで再確保が起きて、どこでヒープに逃げて、なぜその実装が遅くなったり壊れたりするのか、そこまで腹落ちしているかというと別問題です。

実務で使う Go は、単独の言語機能として存在しているわけではありません。メモリ、OS、I/O、ネットワーク、永続化、並行処理、プロセス運用といったテーマの上に乗っています。

つまり、本当に学びたいのは Go そのものだけではなくて、Go を通して触れる実装の原理のほうなんだと思います。

だからこのカリキュラムでは、Go をひとつの言語として閉じずに、`システムを作るための実装技術` として学べるように設計しました。

もう少し正直に書くと、自分の中に「Go の教材を読んでも、たまにきれいに分かれすぎていて逆に現実感が薄い」という感覚がありました。実際の実装では、メモリの話と I/O の話と設計の話と運用の話は、そんなにきれいに切り離されていません。だからこそ、最初から最後まで一本の流れで見られる形が必要でした。

## このカリキュラムの特徴

### 1. Goを「言語」ではなく「実装の道具」として学ぶ

このカリキュラムでは、学習対象を文法や標準ライブラリだけに限定していません。メモリ、OS、I/O、並行処理、ネットワーク、DB、運用まで含めています。

これは「範囲を広げたいから広げた」というより、むしろ Go をちゃんと理解しようとすると自然にそこまで届いてしまう、という感覚に近いです。

たとえばポインタを学ぶときも、単に `*` や `&` の使い方を覚えるのではなく、`何がコピーされて、何が共有されるのか` を理解することを重視します。スライスなら、`append` の裏で何が起きるのか、再確保で何が変わるのか、なぜ不用意な共有がバグにつながるのかまで見ます。

並行処理でも、goroutine や channel の書き方だけをなぞるのではなく、なぜ速くなることがあるのか、なぜ簡単に壊れるのか、何が同期で何が非同期なのか、どこで待ち合わせが起きているのかを意識します。

I/O も同じです。ファイルを読めた、HTTP リクエストを送れた、で終わらせるのではなく、なぜ一括読み込みが危険になりうるのか、`io.Reader` と `io.Writer` の抽象化がどれだけ強いのか、`defer` や `context` が単なる書き方の問題ではなくライフサイクル管理の問題なのだ、というところまでつなげて理解したい。

こういう話は、一見すると「Go の勉強」というより「計算機の勉強」や「システムの勉強」に見えるかもしれません。でも、個人的にはそこが面白いところです。Go を学ぶことを通して、言語を超えて残る感覚を育てたい、というのがこのカリキュラムの根っこにあります。

### 2. 第1週から Docker + Web + DB をつなぐ

このカリキュラムでは、第1週のゴールを `Hello, World` にしていません。

ここはかなり意識的にそうしています。もちろん Hello, World は大事ですし、最初に成功体験を作る意味もあります。ただ、最終的に Web アプリや API や業務システムの実装に進むのであれば、もっと早い段階で「最終成果物の輪郭」を見せたほうがいいと思っていました。

なので最初の週から、Docker Compose で環境を立ち上げ、Go バックエンドが DB に接続し、ブラウザからデータを表示するところまでを確認します。

これは難しくしたいわけではありません。むしろ逆で、「この先に学ぶことが、結局どこへつながっていくのか」を最初に共有しておきたいのです。

最初に垂直統合の流れを見ておくことで、そのあとに学ぶメモリ、I/O、並行処理、設計、運用が、全部ばらばらの知識ではなくなります。

Week 2 で値と参照の違いを学ぶときも、「あとで API の実装や性能の話につながるな」と思える。Week 5 で I/O と `context` を扱うときも、「これはサーバの止め方やタイムアウト設計につながるな」と見える。Week 10 のトランザクションも、「DB の話」として孤立しない。

個人的には、この「学習の出口が見えているかどうか」はかなり大きいです。出口が見えないまま基礎を積み上げるのは、どうしても途中で苦しくなります。逆に、最終的にどこへ行くかが見えていれば、地味な週も意味を持ちます。

### 3. 学習順を自由にしない一本道設計

このリポジトリでは、学習順を次のように固定しています。

1. 正本カリキュラムを読む
2. `examples/handson` を `01` から順番に進める
3. `examples/master` を Week 1 から Week 12 まで進める

最近は、自由に拾い読みできる教材の価値がすごく高いですし、自分もそういうものをよく使います。ただ、今回のカリキュラムでは、あえてその作りにしませんでした。

理由は、知識の接続と再現性を重視しているからです。

何かを学ぶとき、自由度が高いことは必ずしも優しさではないと思っています。特に初中級のあいだは、「どれからやるか」を毎回自分で決めること自体がかなり負荷になります。しかも、その判断が正しかったかは、かなり後にならないと分かりません。

たとえば並行処理に先に興味を持って `goroutine` や `channel` に入ること自体は悪くありません。でも、値のコピー、参照共有、ライフサイクル、エラー処理、I/O の性質といった土台が弱いまま進むと、理解が表面だけで止まりやすい。

逆に、最初は少し退屈に見えても、順番通りに積み上げることで後半の理解が急にクリアになることがあります。この感覚は、自分でも何度も経験しました。

だからこのカリキュラムでは、「好きなところから読める」よりも「最後までつながる」を優先しています。

一般的には「興味のある章から読める」「必要なところだけ選べる」構成もありますが、このカリキュラムではそこをあえて採用していません。

既に知っている内容が含まれていても、一度同じ導線を通ることで、後半の内容とのつながりが見えやすくなります。これは効率が悪いようでいて、全体ではむしろ効率が良いことが多いと思っています。

### 4. 完了基準は「書けたか」より「説明できるか」

このカリキュラムでは、各週の完了判定を `コードが書けたか` だけで見ません。`なぜそうなるのかを自分の言葉で説明できるか` を重視しています。

これはかなり大事にしているポイントです。

コードは、サンプルを見ながらならある程度書けてしまいます。今は検索もしやすいですし、補助ツールも充実しています。だから「動くものができた」だけでは、本当に理解したかどうかの判定が難しいことがあります。

逆に、説明できるかどうかを基準にすると、かなりごまかしが効かなくなります。

たとえば次のような問いに答えられるかを基準にします。

- なぜ Go から DB へ `localhost` ではなくコンテナ名で接続するのか
- なぜポインタレシーバを選ぶのか
- なぜ巨大ファイルを一括読み込みしない方がよいのか
- なぜ `count++` は並行処理で安全ではないのか
- なぜトランザクションなしでは整合性が壊れるのか

こういう問いに、自分の言葉で、相手に伝わる形で答えられるなら、その理解はかなり強いです。

もちろん、全部を最初から完璧に説明できる必要はありません。でも、少なくとも「そう書くものだから」ではなく、「こういう仕組みだから、こう書く」と言える状態を目指したい。

手を動かして動作確認することと、仕組みを言葉で説明できること。この両方がそろって、初めて学習が定着すると考えています。

## 12週間の全体像

全体は次の12週で構成しています。

| 週 | テーマ | ゴール |
| --- | --- | --- |
| 1 | Docker ベースの開発基盤 | Web、DB、Front の E2E 疎通を確認する |
| 2 | メモリ管理と動的データ | 値、参照、スライス、文字列の物理的挙動を理解する |
| 3 | データ集約とカプセル化 | 構造体、メソッド、可視性で安全な境界を作る |
| 4 | 抽象化と疎結合設計 | interface と依存性制御で変更に強い構造を作る |
| 5 | I/O とリソース管理 | Reader/Writer、defer、context で外部資源を扱う |
| 6 | 品質保証と性能計測 | テスト、ベンチマーク、解析で正しさと性能を証明する |
| 7 | 並行処理の基本原理 | goroutine、channel、select を理解する |
| 8 | 高度な並行設計 | Mutex、WaitGroup、Worker Pool で実戦設計する |
| 9 | Web プロトコルと通信 | HTTP、JSON、ミドルウェアで API を構築する |
| 10 | 永続化と整合性 | database/sql、トランザクション、接続管理を扱う |
| 11 | システムアーキテクチャ | レイヤ分離、エラー戦略、ログ設計を固める |
| 12 | 配布最適化と運用 | scratch、静的リンク、ヘルスチェック、終了制御を整える |

こうやって一覧にすると整然として見えるのですが、実際に自分の中で重視しているのは、各週のテーマ名そのものよりも、前後のつながりです。

前半は、Go の土台と実装原理を固める期間です。値と参照、スライス、文字列、構造体、可視性、依存の向き、I/O、リソース管理、テスト、性能計測。このあたりは派手ではありませんが、後半の実装で効いてくる基礎体力です。

中盤では、並行処理や HTTP、JSON、DB といった、いわゆる「実務っぽい」テーマに入っていきます。ただ、ここに来る前に基礎が積んであることで、単なる機能学習ではなく、設計や振る舞いとして理解しやすくなります。

後半では、アーキテクチャ、エラー戦略、ログ設計、配布、ヘルスチェック、終了制御といった、作ったあとに必要になるテーマを扱います。ここを最後まで含めることで、ようやく「動くコード」から「保てるシステム」へ視点が移るようにしたいと思っています。

ここで意識しているのは、`文法を終えてから実務へ進む` という分断を作らないことです。最初から最後まで、常に「最終的に動くシステムへどう接続するか」を見据えています。

## なぜこの順番にしているのか

このカリキュラムの順番には、かなりはっきりした意図があります。

最初に Docker と Web と DB の疎通を置いているのは、学んだことの出口を早い段階で見せるためです。最終成果物の輪郭が見えていると、その後のメモリや I/O や並行処理の学習が、単なる知識の断片ではなくなります。

個人的に、学習の途中で一番しんどいのは、「この勉強は何につながるんだろう」と感じる瞬間です。そこをできるだけ早く越えられるようにしたかった。

そのうえで、Week 2 から Week 6 では、メモリ、構造化、I/O、テスト、性能計測といった土台を固めます。この部分は地味に見えますが、後半で API や DB や並行処理を扱うときの判断力に直結します。

たとえば、性能の話をするなら、単に「速い実装を覚える」では足りません。どこでコピーが起きるのか、どこでアロケーションが増えるのか、どこでストリームにすべきなのか、といった視点が必要になります。だからこの順番です。

Week 7 以降で並行処理、HTTP、JSON、DB を扱うのは、先に基礎的なデータの扱いと依存関係の整理を学んでおくことで、複雑さを受け止めやすくするためです。

そして最後にアーキテクチャと運用を置いているのは、動くものを作ったあとに、`どう保守するか` `どう安全に止めるか` `どう配布するか` まで含めてシステムとして完成させたいからです。

ここは、個人的にかなり大事です。アプリケーションは起動したら終わりではありません。止め方、ログの残し方、障害時に追えるかどうか、ヘルスチェックをどう出すか、そういう話まで含めてようやく実装です。

教材によっては、そこまで行く前に終わることも多いです。でも、自分としてはそこを切り落としたくありませんでした。むしろ最後の最後で「運用まで含めて考えると、ここまでの知識が全部つながる」という感覚を持てるようにしたかった。

## どんな人に向いているか

このカリキュラムは、次のような人に向いています。

- Go の入門は一通り触ったが、その先に何を学ぶべきか曖昧な人
- Web、DB、並行処理、運用までを別々ではなくつながった形で理解したい人
- 仕様や原理を押さえながら、実際に動く成果物まで作りたい人
- 学習順を自分で毎回判断するより、ある程度整理された導線に乗りたい人
- 一発で覚えるより、少し遠回りでも崩れにくい理解を作りたい人

特に相性がいいのは、「断片的には知っているけれど、まだ一本につながっていない」という人だと思います。

Go の本を何冊か読んだ。`net/http` を使ったサンプルも書いた。Docker も少し触った。SQL も知っている。だけど、それぞれが別の箱に入っていて、ひとつの開発体験としてはまだつながっていない。そういう状態の人には、かなり意味のある導線になるはずです。

逆に、文法の一部だけをつまみ食いしたい人や、必要な箇所だけ断片的に参照したい人には、少し重たく感じるかもしれません。

それでも、腰を据えて Go を学びたい人にとっては、かなり相性のよい構成になると思っています。少なくとも、自分が「こういう順番ならもっと腹落ちしやすかった」と感じるものには近づけたつもりです。

## 学習の進め方

学習の流れは、次の3段階です。

```text
examples/handson  ->  examples/master  ->  examples/appendix
基礎文法編             実践編（12週）         付録
```

まず `examples/handson` で、値、型、制御構文、関数、ポインタ、構造体、interface、並行処理、I/O、テスト、HTTP、JSON、DB の基礎を順番に固めます。

ここは、いわゆる「基礎文法編」ですが、個人的には単なる入門ではなく、後半に進むための足場だと思っています。だから順番も固定していますし、既に知っている内容があっても、通しで進めることに意味があります。

そのあと `examples/master` で、12週間の実践編に進みます。ここでは Docker、Web、DB、計測、アーキテクチャ、運用までを一続きの流れとして扱います。

この「基礎文法編から実践編へつなぐ」という構成は、自分の中ではかなりしっくり来ています。最初から重すぎず、でも文法だけで止まらない。基礎を土台としてちゃんと回収していける流れだからです。

付録の `examples/appendix` は、本編を終えたあとに進む発展モジュールです。Fyne による GUI や Web サーバ全体像など、本編の知識を別の舞台に応用する位置づけです。

ここも少しこだわったところで、付録はあくまで本編の外に置いています。面白いテーマを最初から大量に混ぜると、学習導線がまた散ってしまうからです。まずは本編で Web、API、DB、並行処理、運用の軸を固める。そのあとで、GUI や別テーマに広げる。この順番にしています。

## この先 note でどう公開していくか

ここまで書いてきたのは、あくまでカリキュラム全体の考え方と骨格です。これは無料で公開しておいた方がいいと思っています。なぜなら、このカリキュラムが何を目指していて、どういう順番で進むのかは、学ぶ人が最初に見渡せる状態であるべきだからです。

一方で、実際に学習を進めるときに本当に効いてくるのは、各週の細かい補助線です。

たとえば、

- どこでつまずきやすいのか
- 何を誤解しやすいのか
- どこまで理解できればその週を終えてよいのか
- 実務ではその知識がどこに効いてくるのか
- handson や master をどう読むと理解しやすいのか

といった部分は、全体紹介の記事だけではどうしても書ききれません。

なので note では、全体像や学習思想は無料で開きつつ、各週の詳細解説は一部有料で展開していく形が自然かなと思っています。

無料記事では、「このカリキュラムは何か」「なぜこの順番なのか」「自分に向いているか」を判断できるところまでをしっかり書く。有料記事では、「どう理解して、どう進めればよいか」をもっと具体的に掘り下げる。そんな役割分担を考えています。

これは単純に情報を隠したいからではなく、無料部分では骨格をちゃんと渡し、有料部分では伴走に近い価値を乗せたいからです。

個人的には、有料化するなら「続きが読めます」よりも、「理解が深くなって、進みやすくなります」のほうが誠実だと思っています。なので、もし有料で出すとしても、単なる続きではなく、各週の深掘り、つまずきポイント、実務との接続、確認観点を中心にしていくつもりです。

## このカリキュラムで本当にやりたかったこと

ここまで書いてきたことを、もう少し個人的な言い方でまとめると、このカリキュラムでやりたかったのは「Go を勉強した人」を増やすことではなく、「Go で実装を考えられる人」に近づくための道筋を作ることでした。

文法の理解はもちろん大切です。ただ、現実の実装では、文法だけでは判断できないことばかり出てきます。

このデータはコピーしてよいのか。
ここはポインタで持つべきか。
この I/O は一括で読むべきか、ストリームで流すべきか。
この処理は goroutine に逃がすべきか、同期で持つべきか。
この DB 操作はトランザクションで囲わないと何が壊れるのか。
このアプリはどう止まるべきか。

こういう問いに向き合うとき、必要なのは「Go の機能一覧」ではなく、仕組みをつないで考える力だと思っています。

だからこのカリキュラムでは、各トピックを単発で終わらせず、前後のテーマと接続する形にしています。ポインタの話が性能の話につながり、I/O の話が HTTP や運用につながり、interface の話がテストやアーキテクチャにつながる。そういう一本の線を作りたかった。

正直、こういう構成は短くきれいにはまとまりません。寄り道もありますし、地味な週もあります。でも、地味なところを飛ばしてしまうと、後半の理解が薄くなりやすい。そこは今回はかなり意識して、派手さより土台を優先しました。

## まとめ

このカリキュラムで目指しているのは、Go の文法をなぞれるようになることではありません。

メモリ、I/O、並行処理、Web、DB、運用まで含めて、`なぜその実装になるのか` を理解しながら、Go でシステムを組み立てられる状態を目指しています。

もし Go 学習が `文法の断片` のまま止まっている感覚があるなら、こうした一本道のカリキュラムはひとつの有効な形だと思います。

そして、もし自分と同じように「文法の先にある実装感覚をどう育てるか」に引っかかっている人がいるなら、この構成はきっと役に立つはずです。

今後は、この正本カリキュラムを軸にしながら、各週の教材、実践ハンズオン、補助ガイドをさらに育てていく予定です。一本の導線としての完成度を上げつつ、読むだけでなく、手を動かしながら理解できる形にしていきたいと思っています。

note でも、この全体像の記事を入口にしながら、少しずつ各週の詳細を書いていく予定です。まずは Week 1 から、なぜ Docker・Web・DB を最初にまとめて触るのか、その理由と学び方をもう少し具体的に掘っていきます。
