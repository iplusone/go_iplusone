# Go実装技術マスター note公開戦略

このドキュメントは、`Go実装技術マスター 12週間カリキュラム` を note で段階的に公開していくための設計メモです。

## 基本方針

このカリキュラムは、単発記事として売るよりも、`連続して追う意味がある体系` として見せた方が強いです。

そのため、note では次の考え方を採用します。

- 無料記事で `思想` と `全体像` を公開する
- 有料記事で `進め方` と `深い理解` を提供する
- 無料部分だけでも価値がある状態にする
- 有料部分は「続きを読むための課金」ではなく「理解を深めて進めやすくするための課金」にする

## 無料記事の役割

無料記事で担う役割は次のとおりです。

- このカリキュラムが何を目指しているかを伝える
- なぜこの順番なのかを納得してもらう
- 読者が「これは自分向けか」を判断できるようにする
- 有料部分に進まなくても、全体像は持ち帰れるようにする
- 筆者の考え方や信頼感を伝える

無料記事では、読者に「売られている」と感じさせるよりも先に、「この人はかなり考えて作っているな」と思ってもらうことが大切です。

## 有料記事の役割

有料記事で担う役割は次のとおりです。

- 週ごとの理解を深める
- つまずきやすいポイントを先回りして解消する
- 実務とどうつながるかを具体化する
- 何をどこまで理解すればよいかを明確にする
- handson や master を読み解く補助線になる

有料部分で売るべきなのは、情報量そのものより、`迷わず進めるための整理` と `理解の解像度` です。

## 無料と有料の切り分け

### 無料で出すもの

- カリキュラム全体の思想
- 12週間の全体像
- 各週で何を学ぶのかの要約
- なぜこの順番なのか
- このカリキュラムが向いている人、向いていない人
- 学習の基本ルート
- 進捗報告、制作メモ、更新ログ

### 有料で出すもの

- 各週の詳細解説
- 重要概念の噛み砕いた説明
- 学習時によく混乱するポイント
- 実務でどこに効くかの接続
- 演習課題や確認ポイント
- 「説明できる状態」に達するための問い
- handson / master の読み解きガイド
- 実装時の設計判断の補足

## note公開用の無料記事構成

最初の無料記事は、カリキュラムの宣言記事として使います。

### 記事タイトル案

- Goの文法学習で終わらせない、12週間の実装技術カリキュラムを作った
- Docker・Web・DB・運用までつなげて学ぶ Go 実装技術マスター
- なぜこのGoカリキュラムは一本道なのか

### 無料記事の目的

- カリキュラムの存在意義を伝える
- 読者の悩みに言語化して寄り添う
- 12週間の全体像を見せる
- 次の記事や有料記事へ自然につなげる

### 無料記事の構成

1. 導入
   Go を勉強しても実装感覚が残りにくい違和感から始める
2. このカリキュラムを作った理由
   文法、Web、DB、並行処理、運用が分断されがちな問題を書く
3. このカリキュラムの特徴
   実装技術として学ぶ、第1週から Docker + Web + DB、一本道設計、説明重視
4. 12週間の全体像
   各週のテーマをざっくり紹介する
5. なぜこの順番なのか
   学習順の意図を説明する
6. どんな人に向いているか
   読者に自己判断してもらう
7. この先 note でどう展開していくか
   各週の詳細解説、有料記事、更新予定を伝える
8. まとめ
   カリキュラムの価値を短く締める

### 無料記事の最後に入れるべき一文

無料記事の最後には、次の導線を自然に置きます。

例:

- 次回からは、各週の中身をもう少し具体的に分解して書いていきます
- 各週の詳細版では、つまずきやすい点や実務とのつながりまで含めて整理していきます
- まずは Week 1 から、Docker・Web・DB をどういう順序でつなぐのかを書いていく予定です

## 有料記事の販売設計

### 販売の基本単位

最初は次の3段階で設計するのが扱いやすいです。

1. 単体記事販売
   各週の詳細版を個別販売する
2. 4週ごとのまとめ販売
   `Week1-4` `Week5-8` `Week9-12` の3パックにする
3. 全体マガジン販売
   全12週をまとめたマガジンを作る

### おすすめの販売順

最初から全部有料にするより、次の順が自然です。

1. 無料の全体紹介記事を公開
2. Week 1 の前半無料記事を公開
3. Week 1 の詳細版を有料で出す
4. Week 2 以降も同じ型で展開
5. 4本たまったら `Week1-4パック` を作る

### 価格設計の考え方

価格はここでは固定しませんが、考え方は明確にしておくとぶれません。

- 単体記事は買いやすさ優先
- パックは単体より少し割安にする
- 全体マガジンは継続購入より得になるようにする

読者が払うのは文章量ではなく、`整理された導線` と `迷いを減らす価値` です。価格もその前提で考えます。

## 有料記事のテンプレート

各週の有料記事は、次の型で揃えるとシリーズとして強くなります。

1. この週で何を学ぶのか
2. なぜこの週が必要なのか
3. この週でつまずきやすいところ
4. 用語と概念の整理
5. handson / master をどう読むか
6. 実務ではどこに効くか
7. この週を終えたと判断できる問い
8. 次の週へのつながり

## 各週記事の無料・有料の分け方

### 前半無料

- 今週のテーマ
- なぜそれを学ぶのか
- 全体の中での位置づけ
- よくある誤解の入り口

### 後半有料

- 詳細な概念整理
- つまずきポイント
- 具体的な実装の見方
- 演習や確認観点
- 実務接続
- 学習完了のチェックポイント

この形なら、無料部分だけでも「読む価値がある」と感じられますし、有料部分では「進みやすさ」が明確に上がります。

## noteで避けたいこと

- 無料部分を薄くしすぎる
- 有料部分を単なる続きにする
- 何が買えるのか分からない状態にする
- 体系記事なのに記事ごとの型がばらばらになる
- カリキュラム正本そのものを閉じてしまう

特に最後は重要で、カリキュラムの存在意義そのものは公開しておいた方が信頼につながります。閉じるべきなのは骨格ではなく、`解像度の高い伴走部分` です。

## 記事本文の見直し方針

無料公開用の記事では、次を意識します。

- 思想と全体像をしっかり出す
- 売り込み感を抑える
- ただし「この先どう展開するか」は最後で明示する
- 有料要素の存在は匂わせるだけでなく、役割を明確にする

有料版では、次を意識します。

- 単なる長文化ではなく、理解支援を増やす
- 具体例、判断基準、つまずきポイントを増やす
- 読み終わったあとに「次に何をすべきか」が分かるようにする

## 最初に出すべき3本

1. 無料
   `Goの文法学習で終わらせない、12週間の実装技術カリキュラムを作った`
2. 無料
   `このカリキュラムをどう進めるか: handson から master までの学習導線`
3. 一部無料 + 一部有料
   `Week 1: Docker・Web・DB を最初にまとめて触る理由`

この3本があるだけで、入口、導線、販売導線がひとまず揃います。
