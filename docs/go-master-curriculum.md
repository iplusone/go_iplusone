# Go実装技術マスター 12週間カリキュラム

このドキュメントは、このリポジトリにおける Go 学習カリキュラムの正本です。

目的は、`Go の文法を覚えること` ではなく、`プログラミング全般の知見と計算機の仕組みを理解したうえで、Go で実装できるようになること` にあります。

そのため、このカリキュラムは次の方針で設計しています。

- 学習対象は `言語機能` だけでなく `メモリ` `OS` `I/O` `並行処理` `ネットワーク` `永続化` `運用` まで含む
- 学習の主軸は Docker 上の Linux 環境とする
- 第1週から、Go 単体ではなく `Web + DB + Front` の疎通を前提にする
- 各週は `大項目` `中項目` `小項目` の3層で固定する
- 各項目に `取得目的` `関連知見` `成果物` を対応づける
- Fyne による GUI アプリ開発は本編ではなく付録扱いとする

## 全体方針

### 1. 言語より先に原理を学ぶ

Go は目的ではなく手段です。

習得したいのは次のような普遍的知見です。

- データはメモリ上にどう置かれるか
- OS リソースはどう獲得され、どう解放されるか
- 並行処理はなぜ速くなり、なぜ壊れるのか
- ネットワーク越しの通信は何をやっているのか
- 永続化されたデータの整合性はどう守るのか

### 1-1. 学ぶ人に順番は選ばせない

このカリキュラムは、学ぶ人が都合で順番を組み替える前提では設計していません。

固定ルートは次です。

1. 正本カリキュラムを読む
2. `examples/handson` を `01` から `62` まで進める
3. `examples/master` を週ごとに進める

既知の内容であっても、成果物の接続と再現性のために通過します。

### 2. Docker と Linux を基準にする

ローカルが Windows であっても、学習基準は Docker 上の Linux 環境です。

これにより、次の点を最初から前提にできます。

- パスとファイルシステムは Linux 基準で考える
- Web、DB、Front は別プロセス、別コンテナとして扱う
- `localhost` とコンテナ間通信の違いを理解する
- 配布と実行環境の差異を小さく保つ

### 3. 第1週から垂直統合を確認する

第1週のゴールは `Hello, World` ではありません。

第1週のゴールは次です。

- Docker Compose で環境を立ち上げる
- Go バックエンドが DB へ接続する
- ブラウザからバックエンド経由で DB のデータを表示する

これにより、その後に学ぶメモリ、I/O、並行処理、アーキテクチャが、常に `最終的にどこへ繋がるか` を意識しながら学べます。

## 各週の学習基準

各週の完了判定は、`書けたか` ではなく `説明できるか` を重視します。

最低限、次の問いに答えられる状態をその週の完了とみなします。

- 第1週: なぜ Go から DB へ `localhost` ではなくコンテナ名で繋ぐのか
- 第2週: 値渡しと参照渡しで、何がコピーされるのか
- 第3週: なぜポインタレシーバを選ぶのか
- 第4週: なぜ interface を使うと実装差し替えやテストが容易になるのか
- 第5週: なぜ巨大ファイルを一括読み込みしない方がよいのか
- 第6週: なぜヒープ確保が増えると遅くなりやすいのか
- 第7週: unbuffered channel が同期になるのはなぜか
- 第8週: なぜ `count++` は安全ではないのか
- 第9週: なぜ巨大 JSON はストリームとして扱う方がよいのか
- 第10週: なぜトランザクションなしでは整合性が壊れるのか
- 第11週: なぜ依存方向を一方向に保つべきなのか
- 第12週: なぜ SIGTERM を受けてすぐに kill される設計が危険なのか

## 週ごとのボリューム感

時間は人によって変わるため固定しません。

ただし、目安としては次の考え方を採用します。

- 12週間で1クール完結を前提にする
- 1週間で消化しきれない場合は、次週へ持ち越しても `理解優先` とする
- 合格基準は `他人に仕組みを説明できるか` と `成果物が動くか` の両方とする

## 付録の扱い

### Fyne による GUI 開発

Fyne は本編ではなく付録とします。

理由は次のとおりです。

- 本編は Web/API/DB を通じた現代的なシステム構築を主軸にする
- まず Docker 基準の Linux 環境、HTTP、DB、並行処理を優先して固める
- GUI は、そのあとに学んだ知識を別の舞台へ応用するモジュールとして扱う

付録で扱う場合の候補:

- CGO とネイティブライブラリ
- イベントループと UI 更新
- マルチプラットフォームビルド

## ここから先の整備方針

このドキュメントで `大項目 / 中項目 / 小項目` は固定します。

以後の作業は、この正本に対して次の順で行います。

1. 各週の詳細タスクを別紙で作る

この順序により、構造のぶれを避けながら実装物を増やしていきます。
