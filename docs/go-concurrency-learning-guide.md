## 先に結論

Go の並行処理を学ぶときに、最初に押さえたいのは次の5点です。

1. `go 関数()` で処理を並行に進められる
2. 並行処理では実行順が固定されない
3. 値の受け渡しには `channel` が使える
4. 共有データを直接触るなら `Mutex` が必要になる
5. Go では、共有メモリより `channel` を使った設計の方が整理しやすい場面が多い

## 並行処理と並列処理の違い

この2つは似ていますが、意味は同じではありません。

### 並行処理

複数の仕事を、重なって進められるように設計することです。

重要なのは、`見かけ上同時に進んでいる` ことであり、必ずしも物理的に同時実行される必要はありません。

例:

- ブラウザでページ読み込み中に別の操作もできる
- ひとつのプログラムで通信待ちと画面更新を切り替えながら進める

### 並列処理

複数の仕事を、本当に同時に実行することです。

これは CPU コア数や実行環境の影響を受けます。

例:

- 4 コア CPU で 4 つの計算が同時に走る
- 複数スレッドが別コアで実際に同時実行される

### Go での考え方

Go の `go 関数()` は、まず `並行処理を表現する仕組み` として理解するのが自然です。

その goroutine たちが実際に並列で走るかどうかは、実行環境や Go ランタイムに依存します。

## goroutine の基本

Go では、関数呼び出しの前に `go` を付けるだけで、その関数を別の goroutine として動かせます。

```go
go hello("hello", 50)
hello("bye!", 100)
```

この場合:

- `go hello("hello", 50)` は別 goroutine で実行される
- `hello("bye!", 100)` は main goroutine で実行される

その結果、出力が交互に混ざることがあります。

### 実行順は決まらない

並行処理では、どちらが先に進むかは固定されません。

- 先に `hello` が出ることもある
- 先に `bye!` が出ることもある
- 連続で同じ側が何回も進むこともある

この `順番が不定である` という前提が、並行処理を考えるうえで最も大事です。

## なぜ `time.Sleep` を入れるのか

入門サンプルでは `time.Sleep` がよく使われます。

```go
time.Sleep(time.Duration(n) * time.Millisecond)
```

これは主に次の2つの目的で入っています。

- goroutine の動きが交互に見えるようにする
- main が早く終わりすぎるのを防ぐ

### main が終わると他の goroutine も終わる

Go プログラムは、`main` 関数が終わると終了します。

その時点で、まだ動いている他の goroutine があっても、基本的には一緒に終了します。

そのため、学習用コードでは `time.Sleep` を入れて goroutine が動く時間を確保することがあります。

実務では `time.Sleep` を同期手段として使うより、`channel` や `sync.WaitGroup` のような仕組みを使う方が自然です。

## 共有メモリの考え方

複数の goroutine が、同じ変数を共有して使うことがあります。

たとえば、外側にある `msg` を複数の関数が読み書きすると、両方から同じ値が見えます。

```go
msg := "start"
```

この `msg` に対して:

- main 側が `m0`, `m1` を追加する
- 別 goroutine 側が `h0`, `h1` を追加する

といったことをすると、両者が同じ変数を更新していくため、出力に両方の変更が混ざります。

### 便利だが危険

共有メモリは手軽ですが、複数の goroutine が同時に値を書き換えると次の問題が起きます。

- どの順番で更新されたか分かりにくい
- 途中の状態を別 goroutine が読んでしまう
- 競合によって意図しない結果になる

この問題を避けるための代表的な手段が:

- `channel` で値を受け渡す
- `Mutex` で共有データへのアクセスを保護する

の2つです。

## channel の基本

`channel` は、goroutine 間で値をやり取りするための仕組みです。

### 作成

```go
c := make(chan int)
```

文字列ならこうです。

```go
c := make(chan string)
```

### 送信

```go
c <- 100
```

これは `100 を channel c に送る` という意味です。

### 受信

```go
n := <-c
```

これは `channel c から値を受け取る` という意味です。

`<-` の向きで意味が変わる点が重要です。

- `c <- x`
  `c` に `x` を送る
- `x := <-c`
  `c` から値を受け取る

## channel は同期にも使える

`channel` は単なる値置き場ではなく、goroutine 同士のタイミング調整にも使えます。

### 受信側は値が来るまで待つ

```go
fmt.Println(<-c)
```

このコードでは、`c` に値がまだ送られていなければ、その場で待機します。

つまり:

- 先に受信を書いてもよい
- 値が来るまで止まって待ってくれる

ので、goroutine の終了待ちや計算結果の受け取りに向いています。

### 送信側も相手が必要

基本的な unbuffered channel では、送信にも受信側が必要です。

送る相手がいないまま送信しようとすると、送信側は待ちます。

## `total` の例で見る channel

次のような関数を考えます。

```go
func total(n int, c chan int) {
    t := 0
    for i := 1; i <= n; i++ {
        t += i
    }
    c <- t
}
```

これは:

- 1 から `n` までを足し合わせる
- 最後に結果を channel に送る

という関数です。

呼び出し側はこう書けます。

```go
c := make(chan int)
go total(100, c)
fmt.Println("total:", <-c)
```

この流れは:

1. `total` を goroutine で起動する
2. main は `<-c` で結果を待つ
3. `total` が `c <- t` で値を送る
4. main がそれを受け取って表示する

となります。

## 複数 goroutine から同じ channel に送る

同じ channel に対して、複数の goroutine が結果を送ることもできます。

```go
c := make(chan int)
go total(1000, c)
go total(100, c)
go total(10, c)

x, y, z := <-c, <-c, <-c
fmt.Println(x, y, z)
```

このとき重要なのは、`受け取る順番は起動順ではない` ことです。

先に値を送ってきた goroutine の結果から受け取られます。

そのため、出力は環境によって変わります。

例:

- `55 5050 500500`
- `500500 55 5050`
- `5050 500500 55`

### なぜ順番が変わるのか

goroutine ごとに処理時間が違うからです。

`total(10, c)` は `total(1000, c)` より早く終わる可能性が高いため、後から起動したとしても先に値を送ることがあります。

## deadlock とは何か

channel の送受信で、お互いが待ち続けて進めなくなると `deadlock` が起きます。

たとえば、先に送信しようとしているのに、まだ受信側が存在しない場合です。

```go
c := make(chan int)
c <- 100
go total(c)
```

このような形では、main が `c <- 100` のところで待ってしまい、`go total(c)` まで進めません。

結果として、誰も先に進めず停止します。

### 回避の基本

受け取る goroutine を先に動かしておくことです。

```go
c := make(chan int)
go total(c)
c <- 100
```

この形なら:

- `total` 側は `<-c` で待つ
- main 側が `c <- 100` で値を送る

ので、正常にやり取りできます。

## 双方向のやり取りは channel を分けると分かりやすい

main から値を送り、goroutine から結果を返してほしいことがあります。

その場合は channel を2本に分けると整理しやすいです。

```go
cs := make(chan int)
cr := make(chan int)
go total(cs, cr)

cs <- 100
fmt.Println("total:", <-cr)
```

この例では:

- `cs`
  main から total へ送る channel
- `cr`
  total から main へ返す channel

という役割になります。

ひとつの channel を無理に双方向で使うより、`どちら向きの通路か` を分けた方が読みやすくなります。

## goroutine 同士で channel を使う

channel は main と goroutine の間だけでなく、goroutine 同士でも使えます。

たとえば:

- `first` が文字列を送る
- `second` がその文字列を受け取って表示する

といった構成です。

この形の利点は、共有変数を直接触らずに済むことです。

つまり:

- `first` は送ることだけ考える
- `second` は受け取って処理することだけ考える

というように責務を分けやすくなります。

## channel は値が届くまで待つ

Go の channel 理解でとても重要なのが、この待機動作です。

たとえば:

```go
fmt.Println("total:", <-cr)
```

は、`cr` に値がまだ入っていなければその場で止まります。

そのため:

- `go total(cs, cr)` のあとにすぐ受信を書いてもよい
- 値が準備できたタイミングで再開される

という性質が得られます。

このため channel は、単なるデータ移動だけでなく `同期` にも使えるのです。

## `select` による複数 channel の管理

goroutine が増えると、channel も複数になります。

そのときに使うのが `select` です。

`select` は、複数の channel 操作のうち、今実行可能なものを選んで処理します。

### 基本形

```go
select {
case x := <-c1:
    fmt.Println("from c1:", x)
case y := <-c2:
    fmt.Println("from c2:", y)
case z := <-c3:
    fmt.Println("from c3:", z)
default:
    fmt.Println("no value")
}
```

### `switch` との違い

- `switch`
  値を見て分岐する
- `select`
  channel 操作の準備状態を見て分岐する

### `select` の挙動

- 受信できる case があれば、その case が実行される
- 複数の case が同時に実行可能なら、その中からひとつが選ばれる
- どれも実行できず `default` があれば、`default` が実行される
- `default` がなければ、どれかが実行可能になるまで待つ

### `count` の例

3つの goroutine が、それぞれ別 channel に数字を送る例では、main 側で `select` を使って:

- `c1` なら `* first`
- `c2` なら `** second`
- `c3` なら `*** third`

のように出力を振り分けられます。

これにより、`どの channel から値が来たかに応じて処理を変える` ことができます。

## 共有メモリを守る `Mutex`

channel ではなく、同じ変数を複数 goroutine で共有したい場合があります。

そのとき使うのが `sync.Mutex` です。

### 役割

`Mutex` は、ある処理の間だけ `他の goroutine はこの共有データに触れないでほしい` という制御を行います。

### 基本操作

```go
mu.Lock()
// 共有データにアクセスする
mu.Unlock()
```

- `Lock()`
  ロックを取る
- `Unlock()`
  ロックを解放する

### 構造体にまとめる例

```go
type SrData struct {
    msg string
    mux sync.Mutex
}
```

このようにすると:

- `msg`
  共有したいデータ
- `mux`
  そのデータを守るためのロック

をひとつにまとめられます。

## `Mutex` を使うと何が起きるか

`hello` と `main` の両方が同じ `msg` を更新する例で、それぞれの `for` の前で `Lock()` し、最後に `Unlock()` すると、片方の goroutine が処理を全部終えるまで、もう片方は待たされます。

結果として:

- 先に `hello` が全部進む
- そのあとで `main` が全部進む

というような、ほぼ直列処理に見える出力になります。

これは `Mutex` が正しく効いている証拠でもありますが、同時に `ロック範囲が広すぎる` 例でもあります。

## ロックは最小範囲で使う

`Mutex` は便利ですが、広い範囲をずっとロックすると、他の goroutine が何もできなくなります。

それでは並行処理の利点が薄れてしまいます。

大切なのは:

- 本当に共有データに触る瞬間だけロックする
- 計算や待機まで全部ロックしない

という設計です。

避けたい例:

- `for` 全体をロックする
- `time.Sleep` している間もロックしたままにする

目指したい考え方:

- 更新が必要な短い区間だけ `Lock`
- 終わったらすぐ `Unlock`

## channel と Mutex の使い分け

この2つは競合するものではなく、目的が少し違います。

### channel が向いている場面

- goroutine 間で値を受け渡したい
- 実行順や待ち合わせを自然に表現したい
- 処理の責務を送り手と受け手で分けたい

例:

- ワーカーに仕事を渡す
- 結果を集める
- 複数入力のうち、先に来たものを処理する

### Mutex が向いている場面

- 同じメモリ上のデータを複数 goroutine で共有したい
- 状態をひとつの構造体として持ちたい
- 値の受け渡しより `共有状態の保護` が中心課題である

例:

- カウンタの更新
- キャッシュや共有マップの保護
- 構造体フィールドの整合性維持

### 学習段階での指針

迷ったら、まずは `channel で値を渡す設計` を考えると整理しやすいです。

どうしても共有状態を直接持つ必要があるときに `Mutex` を使う、という理解が実用的です。

## この章の学習順

並行処理を学ぶときは、次の順で理解するとつながりやすくなります。

1. `go` で goroutine を起動する
2. `time.Sleep` を使った簡単な動作確認
3. main 終了で goroutine も終わることを知る
4. 共有メモリのサンプルで、順番が不定だと確認する
5. `channel` で送信と受信を学ぶ
6. `deadlock` を体験して、送受信が相手待ちだと理解する
7. 2本の channel で双方向通信を学ぶ
8. `select` で複数 channel を扱う
9. `Mutex` で共有メモリを保護する
10. ロック範囲を最小化する重要性を理解する

## 学習時によく混乱するポイント

### 1. `go` を付けたら必ず同時実行されるわけではない

`go` は並行処理を作りますが、見た目の実行順は保証しません。

### 2. `time.Sleep` は同期の本質ではない

学習用には便利ですが、goroutine 間の正しい連携は `channel` や `sync` 系で考えるべきです。

### 3. channel はただの箱ではない

送受信のタイミングそのものが同期に使われます。

### 4. 複数 goroutine の結果順は起動順ではない

処理時間やスケジューリングで順番は変わります。

### 5. `Mutex` を使えば安全、では終わらない

ロック範囲が広いと性能や並行性が悪化します。

## 最後に覚えておきたいこと

Go の並行処理は、`とにかく同時に動かす技術` というより、`順番が決まらない処理を安全に組み立てる技術` として捉えると理解しやすくなります。

そのための基本部品が:

- `goroutine`
- `channel`
- `select`
- `Mutex`

です。

最初は:

- 値を渡すなら `channel`
- 共有データを守るなら `Mutex`

という軸で考えるだけでも十分です。

そこから、`deadlock`、`select`、ロック範囲の最小化といった実践的な観点へ進むと、Go の並行処理がかなり読みやすくなります。
