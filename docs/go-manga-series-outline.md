# Go漫画シリーズ構成メモ

このドキュメントは、Go 誕生漫画の続編シリーズを中長期で組み立てるための構成メモです。

第1章は「Go誕生秘話」として公開済みの前提で、その続きとなる第2章以降の役割、重なり整理、各章10話の叩き台をまとめます。

デザイン上の共通ルールは [go-manga-design-guide.md](go-manga-design-guide.md) を正本として参照します。

## 目的

- 漫画シリーズ全体の流れを明確にする
- 各章の役割が重複しないように整理する
- 今後の各話執筆で迷ったときの正本にする

## 全体方針

- 第2章は「Go が何を捨て、何を選んだか」に集中する
- 第3章は goroutine / channel / CSP という並行処理の本丸に集中する
- 第4章は OSS 公開、命名、互換性、コミュニティ形成を扱う
- 第5章は「なぜその時代に Go が必要とされたか」という時代背景を描く
- 第6章は Docker / Kubernetes による Go の決定打を描く
- 第7章は現場が Go を選ぶ実務上の理由を描く
- 第8章は AI 時代における Go の拡張編として扱う
- 第9章は思想と文化の総括に寄せる
- 第10章は未来編として締める

## 章の切り分けで意識すること

### 第2章と第9章

- 第2章は「設計思想そのもの」
- 第9章は「なぜその思想が長く愛されるのか」の総括

### 第5章と第6章

- 第5章はクラウドネイティブ前夜の世界変化
- 第6章はその変化の中で Docker / Kubernetes が Go を押し上げた話

### 第8章

- AI は時事性が強いので、本編の幹より「現代拡張編」として扱う
- 固有サービス名を増やしすぎず、少し抽象化したほうが長持ちする

## 推奨章立て

### 第2章 Go革命

テーマ:
Go 思想が世界へ浸透していく。

推奨する流れ:
「何を捨てたか」から入り、「どう設計したか」「どう書かせたいか」「文化として何が残ったか」へ進める。

1. 継承を捨てる
2. composition の思想
3. interface という答え
4. 賢すぎるコード
5. gofmt の衝撃
6. エラーは隠さない
7. 短く、速く、読みやすく
8. IDE はいらない
9. Go Proverbs
10. シンプルは武器になる

補足:
- `継承を捨てる` `composition の思想` `interface という答え` を連続させると設計思想の芯が通る
- `賢すぎるコード` は clever を嫌う文化論なので、設計説明の後ろに置く
- `IDE はいらない` は挑発的に見えやすいので、公開時は `重武装しなくていい` などに言い換える余地がある
